# 六种多智能体实验结果分析报告

## 1. 报告范围

本报告分析 `outputs/` 下六次实验。每次实验包含 50 道相同编号的题目，分别对应一种设置：

| 输出目录 | 实验设置 |
|---|---|
| `20260716_214959` | `single` |
| `20260716_221249` | `multi` |
| `20260716_225647` | `multi_verifier` |
| `20260716_234539` | `multi_verifier_forced` |
| `20260717_005132` | `multi_candidate_memory` |
| `20260717_014936` | `multi_ask_before_finalize` |

分析依据为各目录中的 `metrics.csv`、`failures.json` 和 `traces_all.json`。准确性来自实验中 DeepSeek Judge 的判定；token 统计包含本地智能体推理和 Judge API 调用。

## 2. 总体结果

| 排名 | 设置 | 正确数 | 准确率 | Oracle gap | 总 token | 平均 token/题 | 相对 Single 成本 | token/正确题 |
|---:|---|---:|---:|---:|---:|---:|---:|---:|
| 1 | `multi_verifier` | 39/50 | **78%** | 5 | 179,291 | 3,585.8 | 3.18× | 4,597.2 |
| 2 | `multi` | 38/50 | **76%** | 5 | 106,670 | 2,133.4 | 1.89× | 2,807.1 |
| 3 | `single` | 36/50 | 72% | 0 | 56,425 | 1,128.5 | 1.00× | **1,567.4** |
| 3 | `multi_candidate_memory` | 36/50 | 72% | 3 | 184,329 | 3,686.6 | 3.27× | 5,120.2 |
| 5 | `multi_ask_before_finalize` | 35/50 | 70% | 4 | 296,318 | 5,926.4 | 5.25× | 8,466.2 |
| 6 | `multi_verifier_forced` | 34/50 | **68%** | 7 | 142,176 | 2,843.5 | 2.52× | 4,181.6 |

核心观察如下：

1. `multi_verifier` 的准确率最高，比 `single` 高 6 个百分点，即多答对 3 题；代价是总 token 增加 122,866，约为 `single` 的 3.18 倍。
2. `multi` 是准确率与成本之间更均衡的设置。它比 `single` 高 4 个百分点，只增加到 1.89 倍 token；每个正确答案的 token 成本也明显低于其余多智能体方案。
3. 候选记忆、强制 Verifier 和最终异议机制均未超过普通 `multi` 或 `multi_verifier`。更复杂的流程没有自动转化为更高准确率。
4. `single` 的准确率不是最高，但成本效率最好。若目标是单位 token 产出，`single` 最优；若优先追求本次样本上的最高正确率，`multi_verifier` 最优。

## 3. 各设置分析

### 3.1 Single Agent

`single` 得到 36/50，准确率为 72%，总计使用 56,425 token。14 个错误全部被归类为“没人会”。这并不表示中间阶段不存在可利用信息，而是该设置只有 Solver A，且 `oracle_gap` 的定义要求中间阶段正确、最终阶段错误；本次输出中没有出现这种情况。

该设置建立了最低成本基线：平均每题 1,128.5 token，每个正确答案约 1,567.4 token。

### 3.2 Multi-Agent

`multi` 得到 38/50，准确率升至 76%。与 `single` 相比多答对 2 题，总 token 增加 50,245。

12 个错误中：

- 7 个为“没人会”；
- 5 个为“finalizer 丢答案”，题号是 6、14、22、29、43。

这说明第二个 Solver 确实增加了正确候选的覆盖率，但 Finalizer 并不能稳定保留已有正确答案。`oracle_gap` 为 5，恰好对应 10% 的题目：正确答案已经在上游出现，却在最终汇总时丢失。这是普通多智能体流程最明确的改进空间。

### 3.3 Multi-Agent + Verifier

`multi_verifier` 得到全组最高结果 39/50，即 78%。相比 `multi` 多答对 1 题，但多使用 72,621 token；边际收益较小。

11 个错误中：

- 6 个为“没人会”；
- 5 个为“verifier 判断错”，题号是 15、27、28、29、48。

50 题中 Finalizer 有 44 题跟随 Verifier 决策、6 题没有跟随。该设置减少了纯粹的 Solver 覆盖不足，但错误瓶颈转移到了 Verifier：当 Verifier 选错或其答案解析错误时，Finalizer 很难恢复。

因此，Verifier 在总体上带来了 2 个百分点的提升，但不是可靠的最终裁决者。更合适的角色是向 Finalizer提供额外证据，而不是拥有不可撤销的决定权。

### 3.4 Multi-Agent + Forced Verifier

`multi_verifier_forced` 只有 34/50，准确率 68%，是六组中最低结果。它比普通 `multi_verifier` 少答对 5 题，虽然节省了 37,115 token，但准确率下降 10 个百分点。

16 个错误中：

- 9 个为“没人会”；
- 7 个为“verifier 判断错”。

其 `oracle_gap` 达到 7，也是六组最高。强制采用 Verifier 决策后，系统失去了 Finalizer 根据完整上下文纠正 Verifier 的机会。结果表明，本实验中的 Verifier 质量不足以承担硬路由或强制裁决职责。

### 3.5 Multi-Agent + Candidate Memory

`multi_candidate_memory` 得到 36/50，与 `single` 持平，但消耗 184,329 token，是 `single` 的 3.27 倍。

14 个错误中：

- 11 个为“没人会”；
- 2 个为“finalizer 不听 verifier”；
- 1 个为“verifier 判断错”。

输出中没有发生候选约束重试或最终回退，说明 Finalizer 形式上都选择了候选表中的答案。约束机制执行成功，但候选集合或候选选择本身不够好，因此没有提高准确率。候选约束只能阻止 Finalizer 生成表外答案，无法保证表内被选答案正确。

### 3.6 Multi-Agent + Ask Before Finalize

`multi_ask_before_finalize` 得到 35/50，准确率 70%，总 token 达到 296,318，是六组中成本最高的设置，也是 `single` 的 5.25 倍。

15 个错误中：

- 11 个为“没人会”；
- 3 个为“verifier 判断错”；
- 1 个为“finalizer 不听 verifier”。

让两个 Solver 再次表达异议增加了大量上下文和生成成本，却没有改善最终结果。与 `multi_verifier` 相比，它多消耗 117,027 token，同时少答对 4 题。当前的异议轮次更像是重复已有观点，而不是提供独立复算或针对性纠错。

## 4. 跨实验失败模式

在六次实验中，以下题目反复失败：

| 题号 | 失败次数 |
|---:|---:|
| 4、5、30 | 6/6 |
| 3、8、22、32 | 5/6 |
| 28、29 | 4/6 |

题号 4、5、30 在所有设置中都失败，说明增加角色和流程无法弥补这些题上的基础求解或答案处理问题。题号 3、8、22、32 也具有高度稳定的失败倾向，应优先用于 prompt、答案抽取和模型能力诊断。

不同设置的错误分类还揭示了一个流程规律：

- 从 `single` 到 `multi`，部分“没人会”转化为“已有正确答案但 Finalizer 丢失”；
- 加入 Verifier 后，错误进一步转化为“Verifier 判断错”；
- 强制 Verifier 会放大这一问题；
- 候选限制和异议轮次并未消除上游判断错误。

换言之，增加智能体提高了候选覆盖率，但系统的主要瓶颈随之从“能否算对”转向“能否正确抽取、判断和保留已有答案”。

## 5. 数据质量与评估风险

### 5.1 六次运行不是严格的单变量消融

脚本默认 `temperature=0.2`，六个设置又是在六次独立进程中运行。因此即使题号相同，Solver A 和 Solver B 的生成也可能不同。例如各实验中的 Solver 正确数并不固定。准确率差异同时包含“架构差异”和“随机采样差异”，不能把 1～2 题的变化全部归因于流程设计。

建议后续使用 `--temperature 0`，或保存并复用完全相同的 Solver 输出，再比较不同聚合策略。

### 5.2 答案抽取会误判包含多个数字的自然语言输出

轨迹中存在推理过程和结论正确，但抽取答案错误的情况。以题号 50 的 `multi_ask_before_finalize` 轨迹为例，Solver 文本明确算出“30 minutes”，但因为没有严格使用 `Final Answer: 30`，抽取逻辑取到了文本末尾附近的“20”，随后 Judge、Verifier 决策和 Finalizer 都围绕错误抽取值运行。

这类问题会同时影响：

- Solver 正确率；
- Verifier 的候选选择；
- `oracle_gap`；
- 最终失败分类。

因此，目前指标衡量的不只是数学求解能力，也衡量模型是否严格遵循最终答案格式。

### 5.3 一份轨迹文件不是有效 JSON

`outputs/20260717_014936/traces_all.json` 无法被标准 JSON 解析器完整读取。错误位于 `solver_objections` 一类包含乱码的字符串附近，疑似存在编码损坏或未正确转义的引号。该实验的 `metrics.csv` 和 `failures.json` 仍可用于汇总，但无法可靠进行完整的字段级统计。

此外，部分中文提示或输出显示为乱码。这可能直接降低本地模型理解“最终异议”任务的能力，也是 `multi_ask_before_finalize` 表现不佳的潜在混杂因素。

## 6. 结论与建议

本次 50 题实验支持以下结论：

1. **最高准确率方案是 `multi_verifier`**：78%，但成本为 `single` 的 3.18 倍。
2. **综合性价比更好的方案是 `multi`**：76%，只比最高结果少 1 题，token 比 `multi_verifier` 少 40.5%。
3. **最低成本方案是 `single`**：72%，每个正确答案的 token 成本最低。
4. **不建议当前版本使用强制 Verifier**：准确率降至 68%，且产生最多的 oracle gap。
5. **候选记忆和最终异议机制暂未证明有效**：两者成本显著上升，准确率分别只有 72% 和 70%。

后续改进建议按优先级排列：

1. 修复中文乱码和 `traces_all.json` 序列化问题，确保实验输入与记录可信。
2. 强制所有角色输出结构化最终答案，或改进答案抽取，避免正确推理被错误数字覆盖。
3. 使用确定性生成或固定 Solver 输出，进行严格的聚合策略消融。
4. 针对 `multi` 的 5 个 oracle gap 改进 Finalizer，使其显式比较所有候选并说明拒绝理由。
5. 保留 Verifier 的建议权，但不要强制采用；对 Verifier 与 Solver 冲突的题目触发独立复算。
6. 重点回归测试题号 4、5、30，以及高频失败的 3、8、22、32、28、29。

在当前数据下，如果部署目标偏重准确率，可选择 `multi_verifier`；如果同时考虑推理成本，`multi` 是更合理的默认方案。
